# 2026-07-09 회고

## 1. 오늘의 리뷰
오늘은 이번 주에 진행했던 미션들과 팀 미션에 대한 전체 피드백(W2 리뷰)을 받았다. W2M4 피드백에서는 empty() 확인 후 get()을 호출하는 방식은 원자적이지 않아서 그 사이에 다른 프로세스가 끼어들 수 있고, get_nowait()은 즉시 예외를 발생시키기 때문에 동시성 환경에서 더 안전하다는 점을 배웠다. try 블록을 어디까지 감쌀지에 따라 코드 구조가 달라진다는 것도 함께 정리했다. W2M5 피드백에서는 웹 스크레이핑 대신 Data API를 활용하는 관점, 샘플링의 목적을 구분하는 것, 그리고 "워드클라우드가 정답이 아니라면?"이라는 질문을 통해 use case 없이 만드는 결과물은 '삽질'이 될 수 있다는 점을 배웠다. "망치를 든 사람에겐 모든 것이 못으로 보인다"는 말처럼, 익숙한 도구가 아니라 문제 해결에 맞는 도구를 선택해야 한다는 관점이 인상 깊었다. 이 외에 AWS 과제를 진행했고, Dockerfile을 Docker Desktop 기능으로 인터랙티브하게 작성하는 방법과 user-data vs 컨테이너 중 무엇을 어디에 둘지에 대한 논의도 있었다. 전반적으로 주어진 것만 수행하기보다 한 걸음 더 나아가 가치를 만들어내려는 노력을 기대하신다는 걸 느꼈다.

## 2. Keep
- 지난 회고에서 Problem으로 적었던 get_nowait()과 get()의 차이를 이번 피드백 시간에 원자성 관점에서 해소했다. 회고에 남긴 의문을 끝까지 가져가서 답을 얻은 흐름을 유지하고 싶다.
- 피드백을 코드 레벨(try 블록 위치, 원자성)과 관점 레벨(use case 중심 사고) 양쪽으로 나눠서 받아들인 점이 좋았다.
- 미션 결과물 자체보다 "이걸 왜 만드는가"를 묻는 질문들을 놓치지 않고 기록해둔 점이 좋았다.

## 3. Problem
- 이번 주 미션들을 요구사항 충족 위주로 수행했고, 그 너머의 비즈니스 가치나 use case까지 스스로 고민하는 데는 부족했다.
- 워드클라우드를 만들 때 "익숙하고 빨리 할 수 있는 방법"을 선택했는지, "문제 해결에 맞는 방법"을 선택했는지 구분하지 못했다.
- 피드백에서 나온 질문들(Docker의 장단점, 여러 EC2에 컨테이너를 배포하려면?)에 대해 내 나름의 답을 정리하지 못하고 들은 것에 그쳤다.

## 4. Try
- 다음 미션부터는 시작 전에 "이 결과물의 use case는 무엇이고, 누가 어떤 문제 때문에 쓰는가"를 한 문단으로 먼저 적고 시작하기
- 피드백에서 받은 열린 질문(여러 EC2에 여러 컨테이너 배포 방법 등)에 대해 내 답을 먼저 작성한 뒤 다음 세션에서 검증받기
- 미션 요구사항을 다 채운 뒤 "여기서 한 걸음 더 나아간다면 뭘 할 수 있을까"를 한 가지씩 추가로 시도해보기
