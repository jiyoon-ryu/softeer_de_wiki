# 2026-07-08 회고

## 1. 오늘의 리뷰
오늘은 Steam 게임 리뷰를 대상으로 ETL 파이프라인을 구축하고 긍정/부정 워드클라우드를 만드는 미션을 진행했다. Extract 단계에서는 Playwright 기반 크롤러(crawler.py)로 긍정/부정 리뷰를 멀티프로세스로 동시에 수집하도록 구성했고, 이미 크롤링한 JSON 파일이 있으면 재크롤링 없이 재사용하도록 force_recrawl 옵션을 두었다. Transform에서는 필요한 컬럼만 남기고, Load에서는 SQLite(meccha.db)에 저장한 뒤 다시 DB에서 읽어오는 흐름까지 확인했다. 전처리에서는 어제와 달리 어퍼스트로피를 유지해서 don't, isn't 같은 부정 축약형을 살렸고, 워드클라우드에서는 collocations=True로 "not fun"처럼 자주 붙어 다니는 두 단어 조합이 반영되도록 했다. 이를 위해 기본 불용어 목록에서 부정어를 제외하고, game·play 같은 도메인 불용어를 추가하는 커스터마이징도 진행했다.

## 2. Keep
- 크롤링, 변환, 저장을 extract/transform/load 함수로 분리해서, 크롤링 로직이 바뀌어도 전체 파이프라인(run_etl)은 건드릴 필요 없게 설계한 점이 좋았다.
- 어제 미션에서 아포스트로피를 삭제했던 것과 반대로, 오늘은 부정 축약형을 살리기 위해 유지했다. 같은 전처리라도 목적(단어 빈도 vs 문맥 표현)에 따라 다르게 판단한 점을 유지하고 싶다.
- collocations=True가 generate()에서만 동작한다는 제약을 파악하고, 그에 맞춰 부정어를 불용어에서 제외하는 등 기능의 동작 원리를 이해하고 사용했다.
- 이미 수집한 데이터가 있으면 재크롤링을 생략하는 옵션을 만들어 불필요한 반복 작업을 줄인 점이 좋았다.

## 3. Problem
- collocation_threshold=10 같은 파라미터 값을 기본 감으로 정했고, 값을 바꾸면 결과가 어떻게 달라지는지 실험해보지 않았다.
- 크롤링을 멀티프로세스로 돌렸지만, 실제로 단일 프로세스 대비 얼마나 빨라졌는지는 측정하지 않았다.
- 워드클라우드에 나온 긍정/부정 표현의 차이를 눈으로 확인하는 데 그치고, 이 게임 리뷰에서 어떤 인사이트를 얻을 수 있는지 해석을 정리하지 못했다.

## 4. Try
- collocation_threshold 값을 바꿔가며(5, 10, 20) 연어가 어떻게 달라지는지 비교하고, 최종 값을 선택한 근거를 마크다운 셀로 남기기
- 크롤링을 단일/멀티프로세스로 각각 실행해 소요 시간을 측정하고, 멀티프로세싱의 효과를 수치로 확인해보기
- 워드클라우드 아래에 "긍정/부정 리뷰에서 눈에 띄는 표현과 그 의미"를 2~3문장으로 해석해 기록하기
