# Word Cloud의 작동 원리

워드클라우드는 텍스트에서 단어 빈도를 계산하고, 빈도가 높을수록 글자를 크게 그리는 시각화다. 
`WordCloud().generate(text)`를 호출하면 내부적으로 빈도 계산 → 크기 결정 → 배치 → 색상 순서로 진행된다.

## 1단계: 단어 빈도 계산

입력된 하나의 긴 문자열을 정규표현식으로 단어 단위로 쪼갠 뒤, 각 단어의 등장 횟수를 계산한다.

- **불용어 제거**: stopwords로 지정된 단어(the, a, is 등)는 집계에서 제외
- **복수형 정규화**:  "cat"과 "cats"는 같은 단어로 합산
- **연어(collocation) 처리**: 기본 옵션 `collocations=True`에서는 "good morning"처럼 자주 붙어 다니는 두 단어(bigram)를 하나의 항목으로 취급한다. 이 경우 구성 단어가 단독으로도, bigram으로도 집계되어 사실상 중복 표시가 발생 -> 과제에서는 collocations=False로 설정

이 단계의 출력은 `{단어: 등장 횟수}` 형태의 빈도 사전이다.

## 2단계: 빈도 → 글자 크기 변환 

빈도 내림차순으로 정렬해 상위 max_words개(과제에서는 200개)만 남기고, 1등 단어를 1.0으로 하는 상대 빈도로 정규화한다. wc.words_로 확인 가능하다.

글자 크기는 빈도에 정비례하지 않고 relative_scaling 파라미터(기본 0.5)가 빈도와 순위를 반반 섞어 결정한다. 빈도에 정비례시키면 1등 단어만 거대해지고 나머지는 안 보이기 때문이다. 이 크기는 확정이 아니라 상한선이고, 배치 단계에서 자리가 없으면 줄어든다.

- `relative_scaling=1.0`이면 크기가 빈도에 정비례한다. 트윗처럼 최빈 단어가 압도적인 데이터에서는 1등 단어만 거대해지고 나머지는 전부 작아진다.
- `relative_scaling=0`이면 순위만 반영한다. 1등과 2등의 빈도 차이가 10배여도 크기 차이는 미미하다.
- 기본값 0.5는 두 방식의 절충으로, 많이 나온 단어가 확실히 크되 하위권 단어도 읽을 수 있도록 만든다.


## 3단계: 캔버스 배치

크기가 큰 단어부터 하나씩 배치한다.

1. 목표 글자 크기로 단어를 렌더링했을 때의 경계 상자(bounding box) 크기를 계산한다. 이때 prefer_horizontal(기본 0.9) 확률에 따라 가로로 눕힐지, 10% 확률로 세로로 세울지도 랜덤하게 정한다.
2. 캔버스에서 무작위 후보 위치를 뽑아, 그 자리에 경계 상자를 놓았을 때 이미 배치된 단어와 겹치는지 검사한다.
3. 겹치지 않는 자리를 찾으면 그곳에 단어를 확정한다.
4. 어떤 후보 위치에서도 자리를 찾지 못하면 글자 크기를 한 단계 줄여서 1번부터 다시 시도한다.
5. min_font_size(기본 4)까지 줄여도 안 되면 해당 단어는 폐기 -> max_words=200으로 설정해도 실제 그려지는 단어가 그보다 적을 수 있는 이유

충돌 검사는 적분 이미지(integral image)로 처리한다. 캔버스의 누적합 테이블을 만들어 두면 직사각형 영역이 비었는지를 모서리 값 4개만 읽어 O(1)에 판정할 수 있다. wordcloud 라이브러리는 이를 IntegralOccupancyMap 클래스로 구현했다.

결과적으로 큰 단어가 중앙의 좋은 자리를 먼저 차지하고 작은 단어들이 틈새를 메우는 형태가 된다.

## 4단계: 색상 결정

배치가 끝난 뒤 colormap(기본 viridis)에서 무작위로 색을 뽑아 입히고 색은 빈도와 무관하다. Greens처럼 밝은 색이 포함된 컬러맵을 쓰면 흰 배경에서 일부 단어가 흐리게 보이는 문제가 생길 수 있고, 컬러맵의 진한 구간만 잘라 쓰면 해결된다.

## random_state의 역할

무작위성은 배치 위치, 글자 방향, 색상 세 곳에 있다. random_state를 고정하면 항상 동일한 그림이 나온다. 시드를 바꿔도 단어 구성과 크기는 그대로이고 배치와 색만 달라진다.


## 한계
워드클라우드는 단순 빈도만 반영하므로 now, today처럼 긍정/부정 양쪽에서 흔한 중립 단어가 두 클라우드 모두에 크게 나타나고, "not good" 같은 문맥도 구분하지 못한다. 클래스 간 차이를 강조하려면 TF-IDF 등으로 빈도를 직접 계산해 generate_from_frequencies()에 넣는 방식으로 개선할 수 있다.
